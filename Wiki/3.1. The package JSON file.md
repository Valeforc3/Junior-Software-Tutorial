## About the Package.JSON file

The package.json file is composed of a few things:
1. Metadata about your project (name, version, which instance it is: dev, prod, etc.)
2. Commands that are ran when your repository hits the server
    > There are two ways of doing these: out of the box scripts, such as "install" or "start" OR a dedicated scripts folder, that is then referenced in the package.json folder.
3. Dependencies (other pieces of software / code that need to be installed for your project to run)
    > Yes, these are the same as "commands that are ran" but I figured they were different enough to require their own dot-point. 

<br><br>
We will keep the project simple for now and use a more basic package.json structure.

#### About the File

I generally format the file from most likely to change, to least likely, which means I use the following order:
1. Metadata (Version numbers, release dates, etc.)
2. Scripts (anything that I need to write a custom execution to solve once the server is started)
3. Dependencies (these are underlying for the server's architecture, and shouldn't change unless I need a newer version of a dependency)

> Since you won't be familiar with which versions of software to use, it's best to either just use the versions in my [package.json](/package.json) file or use the vendor's documentation directly to get the best version.
